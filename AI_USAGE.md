# AI Usage

This document describes how AI assistance was used to build MyNextFavArtist, per the
project's transparency requirements.

## Tools used

- **Claude Code** (Anthropic's agentic CLI, Claude Sonnet 5) — used for essentially the
  entire build: architecture exploration, planning, full implementation across all layers,
  test writing, and on-device debugging/verification.

## What it was used for

**Architecture exploration and planning.** Before writing any code, Claude Code explored the
existing [PlanetFinder](https://github.com/JRomCar/PlanetFinder) project (a companion app
built with the same architecture) to understand its module layout, MVI pattern, error-handling
model, navigation setup, and testing conventions. It also researched the MusicBrainz API and
Cover Art Archive directly (rate limits, required headers, endpoint shapes) rather than
assuming API behavior, and verified curated seed-artist MBIDs against the live MusicBrainz
search API instead of guessing them from training data. This research fed into a written
plan (module-by-module porting strategy, explicitly calling out where MusicBrainz's API
shape forces real architecture differences from PlanetFinder — no browse-all endpoint, a
1 req/sec rate limit, no artist-photo API) that was reviewed and approved before
implementation began.

**Scaffolding and boilerplate.** The multi-module Gradle setup (`domain`/`data`/`ui`/`app`/
`test-utils`, version catalog, convention wiring), Hilt DI modules, Room database/DAO,
Retrofit API interface, and DTOs were all generated by porting PlanetFinder's equivalent
files and adapting field names, error variants, and endpoints to MusicBrainz.

**Full feature implementation.** All four screens (Home, Search, Favorites, Details) —
ViewModels, UI actions/effects, Compose screens, and navigation wiring — were implemented by
Claude Code, following the MVI pattern established in PlanetFinder.

**Tests.** ViewModel tests (Home/Search/Favorites/Details), repository and data-source tests,
and two tests with no PlanetFinder equivalent (`RateLimitInterceptorTest`,
`GetHomeArtistsTest` for the seed-query construction) were all written by Claude Code,
mirroring PlanetFinder's existing test structure and stack (JUnit4 + Mockito-Kotlin + Truth).

**Debugging.** Claude Code built and ran the app on an Android emulator, drove it through the
actual UI (tapping through screens, scrolling, reading back screenshots), and used `adb
logcat` to diagnose two real runtime bugs — see below. It also iterated on Gradle/Kotlin
version-compatibility issues (a Coil3/Kotlin metadata mismatch, an AGP built-in-Kotlin/KSP
conflict) that only surfaced when actually running the build, not from code alone.

**Architectural discussion.** Several decisions were made collaboratively via targeted
questions rather than assumed:
- What the Home screen should show, given MusicBrainz has no "trending" or browse-all
  endpoint (a curated seed list was chosen over a favorites-first landing screen or merging
  Home into Search).
- What to name the `Result<D, E>` error wrapper — the plan initially proposed keeping
  PlanetFinder's name (`Outcome`) for maximum fidelity, but the user's prior guidance
  (recorded from an earlier session, for a personal reusable error-handling pattern) was to
  use `Result` when porting this pattern to a new project, so the plan was corrected before
  any code was written.

## What was rewritten or rejected, and why

- **`Outcome` → `Result` naming.** As above: the plan's first draft said "port `Outcome`
  verbatim to match PlanetFinder exactly." This was corrected before implementation once the
  user's standing preference for the reusable pattern's name came up — the whole domain
  layer uses `Result` instead.

- **`BaseUiState` generalization.** PlanetFinder's `BaseUiState.Success` is hardcoded to
  `List<Planet>`, because that app only ever lists one entity type. This project has two
  (artists, and an artist's release groups on the Details screen), so a straight port would
  not have compiled for both use sites. `BaseUiState<T>` was made generic over its payload
  instead of being ported as-is — this was decided during planning, not discovered as a bug,
  but it's the clearest example of "don't blindly port, understand why the original looked
  the way it did."

- **`DetailsUiState` split into three fields instead of one shared `BaseUiState`.**
  PlanetFinder's Details screen has exactly one async concern (the favorite toggle), so one
  shared `BaseUiState` field was enough. This app's Details screen has two independent
  concerns — the favorite toggle and the release-group fetch — and sharing one state would
  mean toggling a favorite blanks the already-loaded album list with a full-screen spinner.
  `DetailsUiState` uses separate `isFavorite`/`isFavoriteActionInProgress`/`releaseGroups`
  fields instead.

- **A broken first draft of `AlbumArtCard`'s error-fallback UI.** While writing the Coil
  image component, an early version tried to build a fallback `Icon` from
  `rememberAsyncImagePainter(model = null)`, which doesn't produce a meaningful placeholder.
  This was caught and rewritten before ever building — it now uses a proper local vector
  drawable (`ic_album.xml`) for the "no cover art available" state.

- **`PullToRefresh` reimplemented on Material3 instead of porting PlanetFinder's Material2
  version.** PlanetFinder uses the older, experimental `androidx.compose.material.pullrefresh`
  API, which needs the separate Material2 dependency. Since this project's Compose BOM
  already ships a stable Material3 `PullToRefreshBox`, the component was rewritten against
  that instead of pulling in an extra dependency to match PlanetFinder's implementation
  exactly.

- **PlanetFinder's global `lateinit var colorScheme` pattern was not carried over.**
  PlanetFinder's `Theme.kt` sets a top-level mutable `ColorScheme` as a side effect of
  composition and reads it from several components (`Shimmer.kt`, `SearchView.kt`,
  `PlanetItem.kt`) instead of using `MaterialTheme.colorScheme`. This is fragile (not scoped,
  set as a composition side effect, not guaranteed populated before first read) and was
  identified as an anti-pattern worth not repeating; every port of a component that used it
  was rewritten to use `MaterialTheme.colorScheme.*` directly.

- **A genuine IPv6-routing bug, found only by running the app.** The Android emulator's
  virtual network resolved an IPv6 address for `musicbrainz.org` that it couldn't actually
  route to, so every request failed with `ConnectException` even though the network was
  otherwise fine. This wasn't something code review would have caught. Fixed with
  `Ipv4PreferringDns`, a custom `okhttp3.Dns` that prefers IPv4 addresses when available —
  this is also a legitimate real-world robustness improvement (broken/partial IPv6 is a real
  issue on some networks, not just emulators), not just an emulator workaround.

  Later removed: intermittent `SSLHandshakeException: connection closed` errors on
  release-group requests persisted on a real device even with this filter disabled, and the
  same URL failed intermittently in the device browser too — confirming the DNS filter was
  never the cause and had no remaining purpose.

- **A real empty-state bug in `FavoritesViewModel`, also found only by running the app.**
  `loadFavoriteArtists()` always set `BaseUiState.Success(data)`, even when `data` was an
  empty list — so a first-time user with zero favorites saw a bare "Delete All Favorites"
  button floating over nothing, instead of the friendly empty-state message. This bug exists
  in PlanetFinder's `FavoritesViewModel` too (it was ported faithfully at first, since
  nothing about the code looked wrong on inspection), but manual testing surfaced it, and it
  was fixed here: an empty result now falls back to `BaseUiState.Initial`, which the existing
  empty-state UI already handles.

## Follow-up review

A separate Claude Code pass (using Opus) reviewed the data layer specifically for latent
bugs rather than porting or building features. Each finding was independently re-verified
against the source before being trusted — one claimed bug (an `InterruptedException` path in
`RateLimitInterceptor`) didn't hold up under verification and was discarded rather than
acted on. The confirmed findings are tracked as follow-up work, not fixed yet.

## What could not be fully verified

Partway through on-device testing, `musicbrainz.org`'s reachability from the test network
degraded (confirmed via `adb shell ping` succeeding over ICMP while TCP:443 connections
consistently timed out or refused to connect, across two independent emulator restarts).
This is external network/API-reachability flakiness, not application code — the same
request/response/error-handling path had already been exercised successfully multiple times
before it started, including a full artist search, a full Details + release-group load with
correct chronological sorting, and correct handling of two different exception types with
working retry. The Search screen's live-query path and the favorite-toggle *write* path
(Room-only, no network involved) were exercised in code and by unit test but not re-confirmed
against the live API after connectivity degraded.
